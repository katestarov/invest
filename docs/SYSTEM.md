# Система оценки инвестиционной привлекательности

## Назначение сервиса

Сервис принимает тикер компании, собирает данные из нескольких внешних источников, приводит их к единому виду, сравнивает компанию с релевантной группой конкурентов и рассчитывает итоговую инвестиционную оценку.

Цель сервиса:

- получить агрегированную картину, а не один показатель
- сделать логику расчета прозрачной
- сохранить путь данных от сырого ответа API до финального score

## Источники данных

### Yahoo Finance

Используется публичный endpoint `chart`.

Сервис берет:

- текущую цену акции
- историю месячных цен
- доходность за 1 год
- доходность за 5 лет
- имя компании

Зачем нужен источник:

- для графика цены
- для рыночного импульса
- для оценки капитализации через цену и число акций

### SEC EDGAR

Используются:

- `company_tickers.json`
- `submissions/CIK...json`
- `companyfacts/CIK...json`

Сервис берет:

- официальное имя компании
- SIC/industry description
- выручку
- операционную прибыль
- чистую прибыль
- активы
- обязательства
- капитал
- текущие активы и текущие обязательства
- операционный денежный поток
- capex
- число акций в обращении

На основе SEC считаются:

- `Debt/Equity`
- `Current Ratio`
- `ROIC`
- `EBIT Margin`
- `FCF Margin`
- `ROE`
- приближенные `P/E` и `P/B`
- история выручки
- история свободного денежного потока

### FRED

Используются ряды:

- `FEDFUNDS`
- `UNRATE`
- `CPIAUCSL`

Сервис берет:

- ставку ФРС
- безработицу
- инфляцию

Особенность:

- нужен `FRED_API_KEY`
- без ключа сервис не падает, а возвращает предупреждение в `warnings`

### World Bank

Используется индикатор роста ВВП США.

Сервис берет:

- рост ВВП США

## Архитектура слоев данных

### Bronze

Таблица:

- `bronze_snapshots`

Содержит:

- тикер
- источник
- сырой JSON
- время получения

Назначение:

- воспроизводимость
- трассировка источника каждого числа
- возможность последующей переобработки

### Silver

Таблица:

- `silver_analyses`

Содержит нормализованные метрики:

- `market_cap_bln`
- `pe_ratio`
- `pb_ratio`
- `roe_pct`
- `roic_pct`
- `ebit_margin_pct`
- `revenue_growth_pct`
- `revenue_cagr_like_pct`
- `fcf_margin_pct`
- `debt_to_equity`
- `current_ratio`
- `one_year_return_pct`
- `five_year_return_pct`
- peer averages
- premium/discount к peer-group

### Gold

Таблица:

- `gold_scores`

Содержит:

- итоговый score
- verdict
- narrative
- готовый payload для UI

## Подбор peer-group

Логика задается в:

- `backend/app/config/peer_groups.json`

Принцип:

1. Сначала ищется правило по `sector`.
2. Затем проверяется совпадение по `industry_contains`.
3. Берется ограниченный список тикеров для сравнения.
4. Если правило не найдено, применяется fallback-группа.

Это важно, потому что мультипликаторы и рентабельность имеют смысл прежде всего в сравнении с похожими компаниями.

## Алгоритм работы сервиса

1. Приходит `GET /api/v1/analyze/{ticker}`.
2. Проверяется TTL-кеш готового анализа.
3. Если кеша нет, сервис идет в внешние источники.
4. Сырые ответы сохраняются в `bronze`.
5. Строится peer-group по конфигу.
6. Считаются нормализованные silver-метрики.
7. По конфигу скоринга считаются блоки оценки.
8. Формируется итоговый score и narrative.
9. Результат сохраняется в `gold`.
10. Готовый `AnalysisResponse` возвращается во frontend.

## Формула скоринга

Файл:

- `backend/app/config/scoring.json`

Блоки:

1. `Рентабельность (Profitability)`
2. `Устойчивость (Stability)`
3. `Оценка относительно сектора (Valuation)`
4. `Рост (Growth)`
5. `Рыночная динамика (Market Momentum)`
6. `Макросреда (Macro)`

### Смысл блоков

`Profitability`

- ROE
- ROIC
- EBIT Margin

`Stability`

- Debt/Equity
- Current Ratio
- FCF Margin

`Valuation`

- premium/discount к peer-group по P/E
- premium/discount к peer-group по P/B

`Growth`

- рост выручки
- динамика выручки
- FCF Margin

`Market Momentum`

- доходность цены за 1 год
- доходность цены за 5 лет

`Macro`

- ставка
- инфляция
- безработица
- рост ВВП

### Нормализация

Каждый показатель приводится к шкале `0-100`.

Используются два типа нормализации:

- `positive`: больше лучше
- `inverse`: меньше лучше

Перед нормализацией значение ограничивается cap-значением из конфига.

### Итоговая оценка

Итоговый score считается как:

- сумма `block_score * weight`

Веса блоков задаются в конфиге и легко меняются без переписывания кода.

## Кеширование

### Кеш провайдеров

Реализован TTL-кеш в памяти процесса для HTTP-ответов.

Что кешируется:

- ответ по связке `url + params`

Настройка:

- `PROVIDER_CACHE_TTL_SECONDS`

Зачем:

- уменьшить число обращений к внешним API
- ускорить повторные запросы

### Кеш анализа

Реализован отдельный TTL-кеш итогового `AnalysisResponse`.

Настройка:

- `ANALYSIS_CACHE_TTL_SECONDS`

Зачем:

- не пересчитывать один и тот же тикер слишком часто
- снизить нагрузку на peer-group запросы

### Сброс кеша

Endpoint:

- `POST /api/v1/cache/clear`

## Что получает frontend

Frontend не считает ничего сам, а получает готовую витрину:

- профиль компании
- итоговый score
- verdict
- narrative
- metric cards
- breakdown блоков
- peer-table
- исторический фундаментальный график
- график цены
- макропанель
- warnings

## Ограничения и особенности

- без `FRED_API_KEY` макроблок будет частично неполным
- SEC может быть чувствителен к `User-Agent`
- часть Yahoo endpoints нестабильна, поэтому фундаментальные коэффициенты опираются на SEC
- peer-group rule-based и может быть расширен вручную

## Почему архитектура сильная

- несколько независимых источников
- понятный ETL-пайплайн
- хранение `bronze / silver / gold`
- конфигурируемая формула
- прозрачный подбор peers
- объяснимый путь данных до итоговой оценки
